Sprint 11.0T - passthroughCount 5-limb debug (S93 audit)

Owner 18:19 root cause confirmed: `curl
212.64.210.85/healthz` works WITHOUT the VPN (the
upstream Patroni answers) but FAILS with the VPN
(the user sees "no route to host" / timeout). The
Sprint 11.0J passthrough (`output.write(buf, 0, n)`)
is NOT actually writing the bytes back to the TUN
output — the kernel drops the packets and the
user's real apps see no internet.

11.0T fix — 5-limb debug + per-write counter:
  1. tun.write() called per read+write +
     passthroughCount increment. New
     `private val passthroughCount = AtomicLong(0)`
     field increments EXACTLY ONCE per successful
     `output.write(buf, 0, n)`.
  2. pfd validity check. Before the write,
     `pfd.fileDescriptor.valid()` is called. If the
     fd is closed (Magisk Zygisk revoke), the reader
     loop exits immediately with Log.e (instead of
     silently failing on the write).
  3. output.flush() immediate. `output.flush()` is
     called immediately after the write (per-packet,
     no batching — confirmed in the code).
  4. DNS UDP 53 detection. The reader inline-checks
     IPv4 protocol (17) + UDP dst port 53 (DNS) or
     853 (DoT) and logs every 50th DNS packet so the
     Owner can grep `adb logcat` to confirm DNS
     queries are reaching the TUN.
  5. passthroughGap breadcrumb. The per-1000-packet
     Log.d now includes both `passthroughCount`
     AND `passthroughGap = packetsObserved -
     passthroughCount`. If `passthroughGap > 0`
     after a `curl 212.64.210.85/healthz` test, the
     write IS being called but the bytes are not
     reaching the kernel (Magisk Zygisk or similar
     fd-revoke interference). If `passthroughGap == 0`,
     every captured packet is also being
     passthrough-written (the healthy state).

  Plus: catch(Throwable) on the write block (in
  addition to the existing catch(IOException)). The
  Owner 18:19 hypothesis is a non-IOException (e.g.
  IllegalStateException on a closed
  AutoCloseOutputStream) which would bubble to the
  outer Throwable handler and log only "TUN reader
  crashed" without the exception class / message.
  11.0T catches the broader Throwable so the Log.e
  shows the actual class + message.
